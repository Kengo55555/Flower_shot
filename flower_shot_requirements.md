## 1. プロジェクト概要

### 1.1 アプリ名
**Flower Shot**

### 1.2 目的
道端に咲いている花や植物をスマートフォンで撮影し、AIが花の名前を判定するアプリ。子どもが「身近な自然への興味」を持ち、撮影記録を通じて達成感を味わえることを目的とする。

### 1.3 ターゲットユーザー
- **メインユーザー**: 小学校低学年（小学1年生〜）の子ども
- **副ユーザー**: 子どもをサポートする保護者
- **管理者**: アプリ管理者（`nomurakengo@gmail.com`）

### 1.4 利用範囲
- **オープン登録 + 事後ブロック方式**
  - URLを知っている人がGoogleアカウントでログインすれば誰でも利用開始可能
  - アプリストアや検索エンジンへの積極的な公開は行わない
  - 不適切な利用が検出された場合、管理者がブロック可能
- **規模想定**: 数十名〜100名程度を想定（API無料枠の範囲内）

### 1.5 利用想定シーン
- 通学路や公園、お散歩中に見つけた花を撮影する
- 撮影記録をマップ上で振り返る
- 月ごとの撮影数を見て「今月は10種類見つけた！」と達成感を得る

## 3. 機能要件

### 3.1 撮影・判定機能

| 項目 | 内容 |
|------|------|
| 撮影方法 | iOS Safariのカメラを起動（`<input type="file" accept="image/*" capture="environment">`） |
| 対象 | 花を中心とした植物（葉・木は第2フェーズで対応検討） |
| 判定方式 | 撮影画像をPl@ntNet APIへ送信し、花の名前を取得 |
| 候補表示 | 信頼度が低い場合は最大3件の候補を提示（信頼度%付き） |
| 判定対象外 | 花以外と判定された場合は「お花が見つからなかったよ」と表示 |
| ネットワーク要件 | 判定処理はオンライン必須。圏外時は「電波のあるところで もう一度ためしてね」と表示 |
| 利用回数制限 | **1ユーザーあたり1日30回まで**（30回到達後は翌日0:00までロック） |

### 3.2 結果表示機能

子どもが読みやすい表示にすることを最優先とする。

- **文字サイズ**: 18pt以上（見出しは24pt以上）
- **使用文字**: ひらがな中心、漢字には必ずふりがな（ルビ）を付与
- **表示内容**:
  - 花の名前（ひらがな表記、例: 「たんぽぽ」）
  - 短い特徴文（30文字以内、例: 「きいろくて まんまるな おはな。はるに さくよ。」）
  - 大きな写真と判定結果アイコン
- **「もっとくわしく」ボタン**: タップで詳細ページに遷移

### 3.3 詳細情報ページ

- Wikipedia API から要約を取得し、アプリ内で表示
- 元記事へのリンクも併記
- 自前で詳細データを持たないため、運用コストゼロ

### 3.4 保存・記録機能

| 保存項目 | 保存先 | 備考 |
|---------|-------|------|
| 撮影画像 | 端末内（IndexedDB） | クラウドへはアップロードしない |
| 花の名前 | クラウド（Firestore） | テキスト |
| 撮影日時 | クラウド（Firestore） | 自動記録 |
| 位置情報（緯度・経度） | クラウド（Firestore） | ユーザーがONにした撮影のみ |
| 判定信頼度 | クラウド（Firestore） | 候補の場合は複数件 |
| ユーザーID | クラウド（Firestore） | Googleログインから取得 |

### 3.5 位置情報の取り扱い

#### 設定方法
- **アプリ設定画面**: 「ばしょをきろくする」のデフォルトON/OFFを設定可能
- **撮影画面**: 撮影ごとに位置情報の取得有無を切り替えできるトグル
- **取得方法**: `navigator.geolocation.getCurrentPosition()`（iOS Safariの標準API）
- **拒否時の動作**: 位置情報なしで保存（撮影自体は可能）

#### マップ表示時の挙動
- 位置情報なしで保存された撮影は、マップには表示されないが図鑑一覧には表示される
- 図鑑一覧には「📍ばしょなし」アイコンを表示

### 3.6 マップ機能

- 撮影した場所を地図上にピンで表示（位置情報ありの撮影のみ）
- ピンをタップすると、撮影写真・花の名前・日付を表示
- 月ごとのフィルタ機能あり
- **使用ライブラリ**: Leaflet + OpenStreetMap（完全無料、APIキー不要）

### 3.7 月間サマリー機能（子ども向けゲーミフィケーション）

子どものやる気を引き出すための機能。

- 月ごとに「今月見つけた花の種類数」を大きく表示
- 累計図鑑コレクション数を表示（例: 「ずかんに 25しゅるい あつまったよ！」）
- バッジ機能（例: 5種類で「はじめての ぼうけんしゃ」、20種類で「はなマスター」）
- アニメーションや効果音で達成感を演出

### 3.8 認証機能（v1.3で改訂）

#### ログイン方式
- **Google アカウントでのSSO**（Firebase Authentication）
- どのGoogleアカウントでもログイン可能（許可リストなし）
- 初回ログイン時に自動的にユーザーレコードを作成

#### アクセス制御（事後ブロック方式）
- 通常時: 全ユーザーがログイン・利用可能
- ブロック時: 管理者が `blocked_users` コレクションにメールアドレスを追加すると、次回ログイン以降アクセス拒否
- ブロックされたユーザーには「ご利用を停止しました。お問い合わせはこちらまで」と表示

#### 新規ユーザー検知
- ログイン時に Firestore の `users` コレクションへ自動登録
- `firstLoginAt` フィールドにタイムスタンプを記録
- `reviewedByAdmin: false` フラグを付与
- 管理者ダッシュボードで未確認ユーザーをハイライト

### 3.9 利用回数制限機能（v1.3で追加）

Pl@ntNet API の無料枠（全ユーザー合計で1日500リクエスト）を保護するため、ユーザー単位での制限を設ける。

#### 仕組み
- Firestore の `apiUsage` コレクションに、ユーザーIDと日付をキーとしてリクエスト数を記録
- 撮影判定実行時、当日のリクエスト数が30以上ならエラー表示
- 翌日0:00（日本時間）でリセット

#### ユーザーへの表示
| 状況 | 表示内容 |
|------|---------|
| 残り10回以下 | 「きょうはあと {N}かい あそべるよ」 |
| 残り0回（上限到達） | 「きょうのお花さがしは おしまい！あしたまた あそぼうね」 |
| 全体上限到達（500/日） | 「きょうはみんなが たくさん あそんでくれたよ！あしたまた あそぼうね」 |

### 3.10 管理者機能（v1.3で改訂）

| 項目 | 内容 |
|------|------|
| アクセス方法 | 管理者専用URL（`/admin`） |
| 認証 | `nomurakengo@gmail.com` のみアクセス可（Firestore Rulesで制御） |
| **ダッシュボード** | 未確認新規ユーザー数、当日のAPI使用量、月次の総撮影数を表示 |
| **ユーザー一覧** | 全ユーザーの一覧、登録日、最終ログイン、撮影数を表示 |
| **新規ユーザー確認** | 未確認ユーザーをワンクリックで「確認済み」にできる |
| **ブロック機能** | 任意のユーザーをブロック・解除できる |
| **撮影記録一覧** | 全ユーザーの撮影記録（テキスト情報のみ）を閲覧可能 |
| 表示しないもの | 撮影画像（端末にしか存在しないため） |

## 4. 非機能要件

### 4.1 性能要件

- 花判定のレスポンス: 5秒以内（API処理時間含む）

- 小1でも操作できるよう、ボタンは大きく・少なく

### 4.3 セキュリティ

- 通信は全てHTTPS
- Firestoreのセキュリティルールで「自分のデータしか読み書きできない」制御
- ブロックされたユーザーは認証直後にチェックされアクセス拒否
- 子どもの位置情報は他ユーザーに公開しない（管理者のみ閲覧）

### 4.4 対応端末

- **iOS 16.4 以降の Safari** を推奨（PWAの安定性とWeb Push対応のため）
- **最低動作要件: iOS 15以降**
- iPhone / iPad 両対応

## 5. iOS Safari の重要な制約と対応

### 5.1 ⚠️ 最重要: ホーム画面追加（PWAインストール）が必須

iOS Safari には **ITP（Intelligent Tracking Prevention）** という機能があり、以下の挙動があります。

- **通常のSafariブラウザでアクセスした場合**: 7日間サイトを訪問しないと、IndexedDB等のローカルストレージが自動削除される
- → 本アプリは **撮影画像をすべてIndexedDBに保存する設計** のため、これに該当すると **撮影写真がすべて消える**

#### 対応策
- 初回ログイン後、必ず「ホーム画面に追加」を案内するチュートリアル画面を表示
- 「ホーム画面に追加」されたPWAは、データが永続化される（ITPの対象外）
- ホーム画面追加されていない状態を検出（`window.navigator.standalone`）して、警告バナーを常時表示

#### チュートリアル例
```
Flower Shot を ながく つかうために、
さいしょに 「ホームがめんに ついか」してね！

📱 [きょうゆうボタン] をタップ
   ↓
📲 「ホームがめんに ついか」をえらぶ
```

### 5.2 IndexedDB の容量制限

- iOS Safari の IndexedDB の上限: **概ね1GB程度**（端末空き容量にも依存）
- 1枚2MBの画像で約500枚まで保存可能

### 5.3 カメラアクセスの仕様

- iOS Safari ではカメラ起動に **ユーザーの操作（タップ）が必須**
- `<input type="file" accept="image/*" capture="environment">` を使うことで、自然なカメラ起動が可能
- 初回利用時、カメラ・位置情報の許可ダイアログが表示される

## 6. 技術スタック（無料運用を前提）

### 6.1 構成図

```
[iPhone / iPad（PWAとしてホーム画面追加済み）]
   ├─ Safari ベースのPWA (Flower Shot)
   │    ├─ <input capture> → カメラ撮影
   │    ├─ Geolocation API → 位置情報（ON時のみ）
   │    └─ IndexedDB → 画像ローカル保存
   │
   └─ ネットワーク（HTTPS）
        ├─ Firebase Hosting（アプリ配信）
        ├─ Firebase Auth（Googleログイン）
        ├─ Firestore（テキストデータ保存・利用回数管理・ブロックリスト）
        ├─ Pl@ntNet API（花の判定）
        ├─ Wikipedia API（詳細情報取得）
        └─ OpenStreetMap（地図タイル）
```

### 6.2 各サービスの選定理由と無料枠

| カテゴリ | 採用技術 | 無料枠 | 選定理由 |
|---------|---------|--------|---------|
| フロントエンド | React + Vite （PWA構成） | 無制限 | iOSへのインストール可能、アプリストア審査不要 |
| ホスティング | Firebase Hosting | 10GB/月 転送、1GBストレージ | Firebaseと統合しやすい |
| 認証 | Firebase Authentication | 月50,000認証まで無料 | Googleログイン標準対応 |
| データベース | Cloud Firestore | 1GBストレージ、読取5万/日、書込2万/日 | テキストのみなら長期間無料運用可能 |
| 画像保存 | 端末内IndexedDB | 端末容量に依存（iOS約1GB） | クラウドストレージ無料枠の枯渇を回避 |
| 花の判定AI | Pl@ntNet API | 1日500リクエスト/組織 | 植物特化、精度が高い、非商用無料 |
| 詳細情報 | Wikipedia API | 完全無料 | 制限ほぼなし |
| 地図 | Leaflet + OpenStreetMap | 完全無料 | APIキー不要 |

### 6.3 Cloud Functions を使わない方針

- Firebase Cloud Functions を使うには **Blazeプラン（従量課金）** への変更が必要（クレカ登録必須）
- 完全無料運用を維持するため、**Cloud Functions は使わず Sparkプラン（無料）に留める**
- 新規ユーザー通知などはサーバーサイド処理ではなく、管理者がダッシュボードで定期的に確認する方式とする

### 6.4 バックエンド選定の根拠【v1.4で追加】

本プロジェクトでは Firebase をバックエンドの中核として採用したが、Supabase等の代替候補と比較したうえでの選定であることを明記する。

#### 検討した候補

| 候補 | 主な特徴 | 採否 |
|------|---------|------|
| **Firebase (Spark)** | Google製のBaaS。認証・DB・ホスティングが統合 | ✅ **採用** |
| Supabase (Free) | OSSベースのBaaS。PostgreSQL・Edge Functions利用可 | ❌ 不採用 |
| Cloudflare (Free) | Workers/D1/R2 など高い無料枠 | ❌ 不採用 |
| PocketBase（自己ホスト） | 軽量なオープンソースBaaS | ❌ 不採用 |

#### Firebase を選んだ決定的要因

##### ① Googleエコシステムとの統合性
- 管理者は `nomurakengo@gmail.com`、ログインも Googleアカウントで運用予定
- Firebase は Google製のため、Googleログインが**最小手順で実装可能**
- 他候補でも Googleログインは可能だが、OAuth クライアント作成・リダイレクトURL登録など追加手順が発生

##### ② Supabaseの「7日無操作で一時停止」問題
本プロジェクトで Supabase を採用しなかった**最大の技術的理由**。

- Supabase 無料プランは、**プロジェクトに7日間データベースアクセスがないと自動的にPause（一時停止）** される
- 再開はダッシュボードから手動操作が必要
- Flower Shot は「お散歩中に思い立って撮影」「冬は使わない」など**散発的な利用パターン**が想定される
- 子どもが久しぶりに使おうとしたら「使えない」状態は致命的
- Firebase Spark には同様の制約がない（常時稼働）

##### ③ iOS PWA運用での実績と日本語の資料量
- iOS Safari の ITP対策、PWAインストール、IndexedDB連携など、本アプリ特有の要件に対する**Stack Overflow・Qiita・公式ドキュメントの量がFirebaseの方が多い**
- トラブル時の検索のしやすさは、個人開発の継続性に直結する

#### 受容するトレードオフ

Firebase（Sparkプラン）採用にあたり、以下の制約は妥協点として受け入れる。

| 妥協点 | 影響 | 代替策 |
|-------|------|-------|
| Cloud Functions が使えない | 新規ユーザー登録時のメール通知ができない | 管理者ダッシュボードで未確認ユーザーをハイライト表示することで代替 |
| Firestore の集計クエリが弱い | 月別の撮影数集計などSQLほど柔軟でない | クライアント側で集計、または事前集計データを別コレクションに保持 |
| ベンダーロックインが強い | 将来の移行コストが高い | データエクスポート機能を将来追加（Phase 3で検討） |

#### 将来的な再評価条件

以下の条件が成立した場合、Supabase等への移行を再検討する。

- ユーザー数が200名を超え、Firestoreの読み書き無料枠を恒常的に超える
- 新規登録通知などのサーバーサイド処理が業務上必須になる
- Firestore のクエリ制約が機能拡張のボトルネックになる

## 7. データ保存方針と制約事項

### 7.1 画像はテキストのみクラウド保存

写真画像は端末内（IndexedDB）にのみ保存。サーバーにはテキスト情報＋位置情報（ON時）のみ保存。

| 保存先 | 内容 | 容量 |
|------|------|------|
| 端末（IndexedDB） | 撮影画像（JPEG） | iOS約1GB（500枚程度） |
| Firestore | 花名、日付、位置、判定信頼度等 | 1GB（事実上無制限） |

### 7.2 ストレージ容量の表示

ユーザー（保護者）が確認できる画面に、以下を**常時表示**する：

```
📸 ほぞんされたしゃしん: 47まい (約95MB / 上限約1GB)
📊 きろく数: 47件 / クラウド使用量 約12KB（無料枠の0.001%）
🌸 きょうの判定: 5かい / 30かい
🏠 ホームがめん追加: ✅ ずみ
```

### 7.3 無料枠超過時の警告

| 警告タイミング | 表示内容 | 対象 |
|--------------|---------|------|
| 端末IndexedDB使用量が800MB（80%）到達 | 「しゃしんを ほぞんできるばしょが すくなくなってきました。古いしゃしんの せいりを おすすめします」 | 保護者 |
| 1ユーザーあたり残り判定回数が10回以下 | 「きょうはあと {N}かい あそべるよ」 | 子ども |
| 1ユーザーあたり判定上限到達（30回） | 「きょうのお花さがしは おしまい！あしたまた あそぼうね」 | 子ども |
| Pl@ntNet API全体が1日400リクエスト到達 | 「もうすぐ きょうの判定が おしまいになるよ」 | 子ども |
| Pl@ntNet API全体が1日500リクエスト到達 | 「きょうはみんなが たくさん あそんでくれたよ！あしたまた あそぼうね」 | 全員 |
| Firestoreの読取/書込が無料枠の80%到達 | 管理者ダッシュボードに警告表示 | 管理者 |
| ホーム画面追加未実施で初回ログイン後3日経過 | 「ホームがめんに ついかしないと しゃしんがきえてしまうかも！」 | 保護者 |

### 7.4 想定運用規模

- ユーザー数: 数十名〜100名程度
- アクティブユーザー数: 1日10〜20名程度を想定
- 1人あたり判定回数: 30回/日上限により、最大でも500回（API枠）以内に収まる
- これを超える場合は、判定上限の引き下げ、あるいはBlazeプラン移行を検討

## 10. データモデル設計

### 10.1 Firestore コレクション構成（v1.3で改訂）

```
firestore/
├── users/                      # ログインしたユーザーの基本情報
│   └── {uid}                   # FirebaseのUID
│       ├── email: string
│       ├── displayName: string
│       ├── photoURL: string
│       ├── firstLoginAt: timestamp     # 初回ログイン日時
│       ├── lastLoginAt: timestamp
│       ├── reviewedByAdmin: boolean    # 管理者が確認済みか（新規検知用）
│       └── settings: object
│           └── locationDefaultOn: boolean   # 位置記録のデフォルト
│
├── blocked_users/              # ブロックリスト（事後ブロック用）
│   └── {email}                 # ドキュメントID = メールアドレス
│       ├── email: string
│       ├── blockedAt: timestamp
│       ├── blockedBy: string   # 管理者メール
│       └── reason: string      # ブロック理由（任意）
│
├── records/                    # 撮影記録（メイン）
│   └── {recordId}              # 自動生成ID
│       ├── userId: string      # FirebaseのUID
│       ├── photoLocalKey: string  # 端末IndexedDBの画像キー
│       ├── flowerName: string  # 確定した花の名前（ひらがな）
│       ├── flowerNameOriginal: string  # APIから返ってきた元の名前
│       ├── candidates: array   # 候補リスト
│       │   └── { name, confidence }
│       ├── confidence: number  # 確定花の信頼度（0-1）
│       ├── capturedAt: timestamp
│       ├── location: object | null   # 位置記録OFFならnull
│       │   ├── latitude: number
│       │   └── longitude: number
│       └── isLocationRecorded: boolean
│
├── apiUsage/                   # API利用回数管理（v1.3で追加）
│   ├── user_{uid}_{YYYYMMDD}   # ユーザー別の日次使用量
│   │   ├── userId: string
│   │   ├── date: string        # "2026-04-30"
│   │   └── count: number       # 当日のリクエスト数（最大30）
│   │
│   └── global_{YYYYMMDD}       # 全体の日次使用量
│       ├── date: string
│       └── count: number       # 当日の全体リクエスト数（最大500）
│
└── admin_logs/                 # 管理者の操作ログ
    └── {logId}
        ├── action: string      # "BLOCK_USER" | "UNBLOCK_USER" | "REVIEW_USER" 等
        ├── targetEmail: string
        ├── performedBy: string
        └── performedAt: timestamp
```

### 10.2 セキュリティルール概要（v1.3で改訂）

```javascript
// 自分のレコードのみ読み書き可、管理者は全閲覧可
match /records/{recordId} {
  allow read: if request.auth.uid == resource.data.userId
              || request.auth.token.email == "nomurakengo@gmail.com";
  allow write: if request.auth.uid == resource.data.userId;
}

// ユーザー情報: 本人と管理者のみアクセス可
match /users/{uid} {
  allow read: if request.auth.uid == uid
              || request.auth.token.email == "nomurakengo@gmail.com";
  allow write: if request.auth.uid == uid
               || request.auth.token.email == "nomurakengo@gmail.com";
}

// ブロックリスト: 管理者のみ書き込み可、本人は自分のエントリだけ確認可
match /blocked_users/{email} {
  allow read: if request.auth.token.email == email
              || request.auth.token.email == "nomurakengo@gmail.com";
  allow write: if request.auth.token.email == "nomurakengo@gmail.com";
}

// API利用回数: 本人のみ書き込み可、管理者は全閲覧可
match /apiUsage/{docId} {
  allow read: if request.auth != null;
  allow write: if request.auth != null;  // 詳細はクライアント側でルール強制
}
```

### 10.3 端末側（IndexedDB）のデータ構造

```
IndexedDB: "flower_shot_db"
└── images（オブジェクトストア）
    └── { key: photoLocalKey, blob: Blob, savedAt: timestamp }
```

- `photoLocalKey` は Firestore の `records/{recordId}.photoLocalKey` と一致
- アプリ起動時にFirestoreから記録一覧を取得し、`photoLocalKey` で端末内画像を引き当てる
- 端末内に画像がない場合（別端末からのアクセス等）は「📷 しゃしんなし」のプレースホルダー表示

## 11. リスクと対応

| リスク | 影響 | 対応策 |
|-------|------|--------|
| ホーム画面追加忘れによるデータ消失 | 撮影写真がすべて消える | 起動時チュートリアル＋常時警告バナー |
| 不正利用者の混入 | API枠の浪費、不適切利用 | 1ユーザー30回/日制限＋管理者によるブロック機能 |
| Pl@ntNet API全体枠の枯渇 | 全ユーザー判定停止 | 1ユーザー上限の動的調整、警告通知 |
| Pl@ntNet APIの仕様変更・廃止 | 判定機能停止 | 代替APIを定期的に評価（Plant.id、Google Vision等） |
| 端末故障で写真が消失 | ユーザーの記録が失われる | 月次エクスポート機能を将来追加 |
| Firestore無料枠超過 | サービス停止 | 管理者ダッシュボードでの監視＋有料移行判断 |
| 判定精度の低さ | 子どもが落胆 | 候補複数表示＋「みつからない時は大人に聞こう」のガイド |
| 子どもが屋外で歩きスマホ | 安全リスク | 起動時に「あんぜんなところで つかおうね」の注意喚起表示 |
| iOS Safariの仕様変更 | 機能不全のリスク | 定期的な動作確認とOSアップデート対応 |
| 個人情報の取り扱い | 法的リスク | プライバシーポリシー掲示＋同意画面の実装 |
