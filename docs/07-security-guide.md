# Flower Shot セキュリティ対策ガイド

> この資料は、セキュリティに詳しくない方が「何が危険で、どう対処したか」を理解するためのものです。

---

## 1. 今回実施した3つのセキュリティ対策

### 対策1: APIキーの保護

| | 内容 |
|---|------|
| 🔴 **元のリスク** | APIキー（外部サービスの合い鍵）がアプリのコードに丸見えで埋め込まれていた |
| 💥 **起きうる問題** | 誰でもキーを盗んでAPIを無制限に呼び出せる → 無料枠（1日500回）が枯渇、最悪アカウント停止 |
| ✅ **実施した対策** | 下表のとおり |
| 🟢 **対策後** | 外部からAPIキーの取得は不可能。プロキシ経由でも自サイト以外は拒否 |

| 対策 | たとえると |
|------|----------|
| APIキーをサーバー側の環境変数に移動 | 鍵をドアマットの下 → 金庫の中に移した |
| Originチェック追加（自サイトのみ許可） | 郵便局に「会員証がないと受付しない」ルールを追加 |
| 転送先を花判定API（/v2/identify/）のみに制限 | 「この住所宛ての荷物以外は受付不可」に制限 |
| POSTメソッドのみ許可 | 荷物の発送のみ、それ以外の窓口業務は停止 |

---

### 対策2: データベースのアクセス制御強化

| コレクション | 🔴 元のリスク | ✅ 対策後 |
|-------------|-------------|----------|
| **apiUsage**（利用回数） | 誰でも任意のカウントを書き換え可能 → 回数リセットし放題 | 自分のUIDのドキュメントのみ、countフィールドのみ変更可 |
| **users**（ユーザー情報） | 「管理者確認済み」フラグを自分でtrueにできる | 更新可能フィールドを`settings`と`lastLoginAt`のみに制限 |
| **records**（撮影記録） | バリデーションなし → 不正データでDB汚染可能 | 必須フィールド検証、confidence範囲(0-1)チェック |
| **（その他全て）** | 未定義コレクションにアクセス可能 | キャッチオールルールで全拒否 |

> 💡 **たとえると**: 図書館の貸出カードを他人が書き換えられる状態 → 自分のカードだけ、決まった欄だけ記入可能に

---

### 対策3: メールアドレスホワイトリスト

| | 内容 |
|---|------|
| 🔴 **元のリスク** | Googleアカウントがあれば誰でもログイン・利用可能（オープン登録） |
| 💥 **起きうる問題** | 見知らぬ人がAPI枠・DB無料枠を消費。子ども向けアプリに不特定多数がアクセス |
| ✅ **実施した対策** | 管理者が許可したメールアドレスのみログイン可能（招待制） |
| 🟢 **対策後** | 管理者画面からメールアドレスの追加・削除が可能。未許可ユーザーは「招待制です」画面で拒否 |

**ログインフロー（3段階チェック）：**

```
Googleログイン
  → ① ホワイトリストに登録済み？  → NO → 「招待制です」画面（アクセス拒否）
  → ② ブロックリストに登録済み？  → YES → 「利用停止しました」画面
  → ③ OK → アプリ利用可能
```

## 2. セキュリティリスクの一般的な解説

### 2-1. APIキーの漏洩

| 項目 | 説明 |
|------|------|
| **APIキーとは** | 外部サービスを使うための「合い鍵」。マンションのカードキーのようなもの |
| **なぜ危険？** | ブラウザのコードは誰でも見える → お店の看板にパスワードを書いているのと同じ |
| **正しい管理** | サーバー側の環境変数に保存。銀行の金庫の鍵は銀行員（サーバー）だけが持つ |

> ⚠️ **実際の事故例**: AWSのキーをGitHubに公開 → 数時間で数百万円の請求。悪意のあるbotが24時間GitHubを巡回してキーを探している

---

### 2-2. データベースのアクセス制御

| 原則 | 説明 | たとえ |
|------|------|--------|
| **最小権限の原則** | 必要最低限の権限だけ付与 | 一般社員は自分のデスクだけ、社長だけが金庫を開けられる |
| **フィールドバリデーション** | データの妥当性を検証 | 「年齢」に「りんご」と書けないようにする |
| **テストモード厳禁** | 開発用の設定を本番で使わない | 玄関の鍵を開けっ放しにしない |

---

### 2-3. 認証と認可

| | 認証（Authentication） | 認可（Authorization） |
|---|---|---|
| **質問** | あなたは誰？ | あなたは何ができる？ |
| **たとえ** | コンサートで身分証を見せる | チケットの種類で座席が決まる |
| **Flower Shotでは** | Googleログイン | ホワイトリスト + Firestoreルール |

**多層防御の考え方（お城の防御）：**

| 層 | 防御策 | Flower Shotでは |
|----|--------|----------------|
| 🏰 お堀 | そもそも入れない | ホワイトリスト（許可メールのみ） |
| 🚪 城門 | 本人確認 | Googleログイン |
| 👀 見張り台 | 問題者を排除 | ブロックリスト |
| 🔑 各部屋の鍵 | アクセス範囲を制限 | Firestoreセキュリティルール |

---

### 2-4. オープンプロキシの危険性

| 項目 | 説明 |
|------|------|
| **プロキシとは** | クライアントと外部サービスの間の「郵便局」。代わりにリクエストを届ける |
| **オープンプロキシとは** | 誰でも使える郵便局。泥棒も使える |

| 悪用シナリオ | 説明 |
|-------------|------|
| API不正利用 | 大量リクエストで無料枠を枯渇させる |
| 踏み台攻撃 | プロキシ経由で他サーバーを攻撃 → あなたが犯人扱いされる |

| 必須の制限 | Flower Shotの設定 |
|-----------|------------------|
| Origin制限 | `flower-shot.pages.dev` と `localhost` のみ許可 |
| パス制限 | `/v2/identify/` のみ転送可 |
| メソッド制限 | POSTのみ |

---

### 2-5. CORS（クロスオリジン）

| 項目 | 説明 |
|------|------|
| **CORSとは** | 異なるWebサイト間のデータやり取りを制御する仕組み |
| **なぜ必要？** | 悪意のあるサイトが裏でネットバンキングに送金リクエストを送るのを防ぐ |
| **「全許可」の危険** | `*`（全オリジン許可）= 家のドアを開けっ放し |

```
CORSの流れ:
  ブラウザ → 「サイトAからBにアクセスしていい？」（プリフライト）
  サーバーB → 「Aは許可リストにある → OK」 or 「拒否」
```

## 3. セキュリティチェックリスト

### APIキー・秘密情報
- [ ] APIキーがブラウザ側のコードに含まれていないか
- [ ] APIキーがGitリポジトリにコミットされていないか
- [ ] `.env`ファイルが`.gitignore`に登録されているか
- [ ] 本番と開発でAPIキーを分けているか

### データベース
- [ ] 「テストモード」のまま本番運用していないか
- [ ] 各ユーザーが自分のデータのみアクセス可能か
- [ ] 更新可能フィールドが最小限に制限されているか
- [ ] 入力データのバリデーションがあるか
- [ ] 未定義コレクションへのアクセスが拒否されているか

### 認証・認可
- [ ] ログイン機能が正しく実装されているか
- [ ] 認証だけでなく認可も設定されているか
- [ ] 管理者権限を自分で付与できない仕組みか
- [ ] ログアウト機能が正しく動作するか

### ネットワーク
- [ ] プロキシにOrigin制限があるか
- [ ] プロキシの転送先が必要なエンドポイントのみか
- [ ] CORS設定が全許可（`*`）になっていないか
- [ ] 全通信がHTTPSか

### 運用
- [ ] エラーログを確認できるか
- [ ] 不正アクセス検知の手段があるか
- [ ] 定期的にセキュリティ設定を見直しているか

## 4. 用語集

| 用語 | 説明 |
|------|------|
| APIキー | 外部サービスの「合い鍵」。持っていればサービスを使える |
| API | ソフトウェア同士がデータをやり取りする窓口 |
| CORS | 異なるWebサイト間のデータやり取りを制御する仕組み |
| Cloudflare | Webサイトの配信・セキュリティを提供するサービス |
| Firebase | Google提供のアプリ開発プラットフォーム（認証・DB等） |
| Firestore | Firebaseのデータベースサービス |
| HTTPS | 暗号化されたWeb通信。URLが`https://`で始まる |
| UID | ユーザー固有のID。Firebase認証で自動割り当て |
| 403エラー | 「アクセス拒否」を意味するエラーコード |
| オープンプロキシ | 誰でも使える中継サーバー。悪用リスクが極めて高い |
| オリジン | Webサイトの「住所」（例: `https://flower-shot.pages.dev`） |
| 環境変数 | サーバーだけが読める設定値。秘密情報の管理に使う |
| キャッチオールルール | 「それ以外は全拒否」するルール |
| クライアント側 | ブラウザで動く部分。誰でも見えるので秘密情報は置かない |
| コレクション | Firestoreのデータグループ（フォルダのようなもの） |
| 最小権限の原則 | 必要最低限の権限だけを付与するセキュリティの基本原則 |
| サーバー側 | ユーザーから見えない裏方。秘密情報はここに保管 |
| 多層防御 | 複数の防御層を重ねてセキュリティを確保する考え方 |
| 認可 | 「あなたは何ができる？」を制御すること |
| 認証 | 「あなたは誰？」を確認すること |
| バリデーション | データが正しい形式・範囲かを検証すること |
| ブラックリスト | 拒否対象を列挙する方式（リストに載っている人を拒否） |
| プリフライト | CORS通信前の「これ送っていい？」という事前確認リクエスト |
| プロキシ | クライアントと外部サービスの間の仲介役（郵便局のようなもの） |
| ホワイトリスト | 許可対象を列挙する方式（リストに載っている人だけ許可） |
