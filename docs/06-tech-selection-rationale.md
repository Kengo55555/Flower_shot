# Flower Shot 技術選定の理由と比較

**この資料の目的**: Flower Shot で使われている各技術が、なぜ他の選択肢ではなくこれに決まったのかを、比較を交えて説明します。

最終更新日: 2026-05-01

---

## はじめに

アプリを作るとき、同じ目的を達成できる技術は何通りもあります。たとえば「地図を表示する」だけでも、Google マップ、Leaflet、Mapbox など複数の選択肢があります。

Flower Shot は **「家族5人程度の小規模利用」「子ども向け」「できるだけ無料」** という条件のもとで、最適な技術の組み合わせを選びました。この資料では、各カテゴリで「なぜこれを選んだのか」を、他の候補と比較しながら説明します。

---

## 1. 画面を作る技術: React vs Vue.js vs Next.js vs Flutter

アプリの画面（ボタン、文字、写真の表示など）をどの技術で作るかの選択です。

| 比較項目 | **React** | Vue.js | Next.js | Flutter |
|---------|-----------|--------|---------|---------|
| 費用 | 無料 | 無料 | 無料 | 無料 |
| 学習資料の多さ | とても多い | 多い | 多い | やや少ない |
| 日本語の情報 | とても多い | 多い | 多い | 普通 |
| 子ども向けアプリへの適性 | 高い | 高い | やや過剰 | 高い |
| 開発の簡単さ | 普通 | 簡単 | やや複雑 | やや複雑 |
| PWA対応 | 可能 | 可能 | 可能 | 可能だが複雑 |
| 他ライブラリとの相性 | とても良い | 良い | 良い | 独自体系 |

### 選んだ理由: React

- **圧倒的な利用者数と情報量**: 世界で最も使われている画面作成技術であり、困ったときに解決策を見つけやすいです。
- **地図（react-leaflet）やFirebase との連携が充実**: Flower Shot で使う他の技術との組み合わせ事例が豊富です。
- **TypeScript との相性が良い**: プログラムの書き間違いを防ぎやすく、安定したアプリが作れます。

### 選ばなかった理由

- **Vue.js**: 優秀な選択肢ですが、react-leaflet のような地図ライブラリの成熟度でReactに劣ります。
- **Next.js**: React をベースにした上位技術ですが、サーバーサイドの仕組み（SSR＝サーバー側で画面を組み立てる機能）が含まれており、今回のアプリには過剰です。
- **Flutter**: Google が作ったアプリ開発技術で、iOS/Android の両方に対応できますが、ウェブアプリとしてはやや不向きで、使用する言語（Dart）の学習コストがかかります。

## 2. 開発ツール: Vite vs webpack vs Create React App

プログラムをアプリの形に変換する道具の選択です。

| 比較項目 | **Vite** | webpack | Create React App |
|---------|----------|---------|-----------------|
| 費用 | 無料 | 無料 | 無料 |
| 起動速度 | とても速い | 遅い | 遅い |
| ビルド速度 | とても速い | 普通 | 普通 |
| 設定の簡単さ | 簡単 | 複雑 | 簡単（制限あり） |
| 最新技術への対応 | 最新 | 対応可能 | 更新停止 |
| 開発中の快適さ | とても高い | 普通 | 普通 |

### 選んだ理由: Vite

- **圧倒的な速度**: プログラムを書き換えてから画面に反映されるまでが一瞬です。開発効率が大幅に上がります。
- **設定がシンプル**: 複雑な設定ファイルを書く必要がほぼありません。Tailwind CSS の導入もプラグインを1行追加するだけです。
- **現在の主流**: React の新規プロジェクトでは Vite を使うのが標準になっています。

### 選ばなかった理由

- **webpack**: 長い歴史がありますが、設定が複雑で、起動・ビルドの速度が Vite に大きく劣ります。
- **Create React App**: かつては React の公式ツールでしたが、2023年に更新が停止されており、新規プロジェクトでの使用は推奨されていません。

## 3. ログイン方法: Firebase Auth vs Auth0 vs 自作認証

「誰がアプリを使っているか」を管理する仕組みの選択です。

| 比較項目 | **Firebase Auth** | Auth0 | 自作認証 |
|---------|-------------------|-------|---------|
| 費用 | 月5万認証まで無料 | 月7,500認証まで無料 | 無料（開発工数大） |
| Googleログイン対応 | 標準で対応 | 対応 | 自分で実装が必要 |
| 開発の簡単さ | とても簡単 | 簡単 | とても難しい |
| 子ども向けの適性 | 高い（Googleアカウント共用） | 高い | 高い |
| セキュリティ | Google が管理 | 専門企業が管理 | 自分で責任を持つ |
| Firestore との連携 | 完璧（同じFirebase内） | 追加設定が必要 | 自分で実装が必要 |

### 選んだ理由: Firebase Authentication

- **Googleログインが最小の手間で実現**: Flower Shot は保護者のGoogleアカウントでログインする設計です。Firebase Auth なら数行のプログラムで「Googleでログイン」ボタンが作れます。
- **Firestore（データベース）との一体運用**: 同じ Firebase の中でログインもデータ保存もできるため、「ログインしたユーザーだけが自分のデータを見られる」というセキュリティルールが簡単に設定できます。
- **5人程度の利用なら完全に無料**: 月5万認証まで無料なので、料金の心配は一切ありません。

### 選ばなかった理由

- **Auth0**: 優秀なサービスですが、Firestore と組み合わせるには追加の設定が必要で、Firebase Auth ほどスムーズではありません。
- **自作認証**: パスワードの管理やセキュリティ対策を自分で行う必要があり、専門知識と大きな開発工数が必要です。セキュリティ事故のリスクも高くなります。

## 4. データベース: Firestore vs Supabase vs PocketBase

花の名前や撮影日時などの文字データを保存する場所の選択です。

| 比較項目 | **Firestore** | Supabase | PocketBase |
|---------|---------------|----------|------------|
| 費用 | 1GB/読取5万回(日)まで無料 | 500MB/5万行まで無料 | 完全無料（自分で運用） |
| 開発の簡単さ | 簡単 | 簡単 | やや複雑 |
| Firebase Auth との連携 | 完璧 | 追加設定が必要 | 別途認証が必要 |
| 無操作時の停止 | なし | 7日無操作で一時停止 | なし（自分で運用） |
| 日本語の情報 | とても多い | 増えている | 少ない |
| 子ども向けの適性 | 高い | 高い | 運用が大人向け |
| サーバー管理 | 不要（Google が管理） | 不要 | 必要（自分で管理） |

### 選んだ理由: Cloud Firestore

- **Firebase Authentication との完璧な連携**: ログイン機能と一体化しており、「自分のデータだけ見える」というルールが簡単に設定できます。
- **無操作で停止しない**: Supabase の無料プランは7日間アクセスがないとデータベースが一時停止してしまいます。Flower Shot は毎日使うとは限らないため、これは大きな問題です。
- **テキストデータだけなら無料枠で十分**: 花の名前、日時、座標などの文字データだけなので、1GBの無料枠を使い切ることはまずありません。

### 選ばなかった理由

- **Supabase**: 機能は優れていますが、7日間無操作で一時停止する仕様が問題です。夏休み中に1週間使わないだけで停止してしまう可能性があります。
- **PocketBase**: サーバーを自分で用意・管理する必要があり、運用の手間とコストがかかります。

## 5. 花の判定AI: Pl@ntNet vs Google Vision vs Plant.id

花の写真を送ると名前を教えてくれるAIサービスの選択です。

| 比較項目 | **Pl@ntNet** | Google Vision | Plant.id |
|---------|-------------|---------------|----------|
| 費用 | 非商用無料（500回/日） | 月1,000回まで無料、以降有料 | 月200回まで無料、以降有料 |
| 植物の判定精度 | とても高い（植物専門） | 普通（汎用AI） | とても高い（植物専門） |
| 日本の花への対応 | 良い（日本語名対応） | 普通 | 良い |
| 子ども向けの適性 | 高い（花に特化） | 低い（花以外も判定） | 高い |
| API制限 | 500回/日（組織全体） | 1,000回/月 | 200回/月 |
| 判定対象 | 植物のみ | 何でも（物体全般） | 植物のみ |

### 選んだ理由: Pl@ntNet API

- **植物に特化した高い精度**: 世界中の研究者と市民科学者のデータに基づいており、花の判定精度がとても高いです。
- **非商用なら無料（1日500回）**: 5人家族で使う場合、1人1日100回の制限を設けても十分な回数です。月額課金ではなく日次リセットなので、翌日にはまた使えます。
- **日本語の花名に対応**: 判定結果に日本語の一般名が含まれており、子どもにもわかりやすい表示ができます。

### 選ばなかった理由

- **Google Vision**: 汎用的な画像認識AIで、花だけでなく動物や建物なども判定してしまいます。花に特化した Pl@ntNet の方が精度が高く、子ども向けアプリに適しています。また、月1,000回の制限は5人で使うと1人あたり月200回しか使えません。
- **Plant.id**: 植物専門で精度は高いですが、無料枠が月200回と少なく、5人で使うと1人あたり月40回しか使えません。有料プランに移行すると月額費用が発生します。

## 6. 地図: Leaflet+OSM vs Google Maps vs Mapbox

アプリ内で地図を表示する技術の選択です。

| 比較項目 | **Leaflet + OpenStreetMap** | Google Maps | Mapbox |
|---------|---------------------------|-------------|--------|
| 費用 | 完全無料 | 月200ドル分の無料枠、以降有料 | 月5万回表示まで無料 |
| APIキーの必要性 | 不要 | 必要 | 必要 |
| 地図の品質 | 良い | とても良い | とても良い |
| 開発の簡単さ | 簡単 | 普通 | 普通 |
| React との連携 | react-leaflet で簡単 | @react-google-maps で可能 | react-map-gl で可能 |
| 子ども向けの適性 | 高い（シンプル） | 高い | 高い |
| オフライン対応 | タイルキャッシュで部分対応 | 限定的 | 限定的 |

### 選んだ理由: Leaflet + OpenStreetMap

- **完全無料でAPIキー不要**: Google Maps は月200ドル分の無料枠がありますが、APIキーの取得にクレジットカード登録が必要です。OpenStreetMap はそもそもAPIキーが不要で、利用料金も一切かかりません。
- **シンプルで軽量**: Flower Shot の地図機能は「花を見つけた場所にピンを表示する」だけなので、高機能なGoogle Maps は不要です。Leaflet で十分です。
- **react-leaflet で React との連携が容易**: React のコンポーネントとして地図を扱えるライブラリが成熟しており、開発がスムーズです。

### 選ばなかった理由

- **Google Maps**: 高品質ですが、APIキーの取得にクレジットカードが必要で、無料枠を超えると課金が発生します。今回の用途にはオーバースペック（過剰な機能）です。
- **Mapbox**: デザインの自由度が高いですが、APIキーが必要で、設定がやや複雑です。

## 7. アプリの公開先: Cloudflare Pages vs Firebase Hosting vs Vercel vs Netlify

作ったアプリをインターネットに公開するサービスの選択です。

| 比較項目 | **Cloudflare Pages** | Firebase Hosting | Vercel | Netlify |
|---------|---------------------|------------------|--------|---------|
| 費用 | 無料（帯域無制限） | 無料（10GB/月まで） | 無料（100GB/月まで） | 無料（100GB/月まで） |
| サーバー処理（Functions） | 10万回/日 無料 | 有料プラン必要 | 無料枠あり | 無料枠あり |
| 表示速度 | とても速い | 速い | とても速い | 速い |
| 帯域幅の制限 | 無制限 | 10GB/月 | 100GB/月 | 100GB/月 |
| デプロイの簡単さ | 簡単 | 簡単 | とても簡単 | 簡単 |
| APIプロキシ | Pages Functions で無料 | Cloud Functions は有料 | Edge Functions で可能 | Functions で可能 |

### 選んだ理由: Cloudflare Pages

- **帯域幅が無制限で無料**: 他のサービスには月あたりの転送量制限がありますが、Cloudflare Pages にはありません。
- **Pages Functions が無料でAPIプロキシに使える**: Flower Shot では、花の判定AI（Pl@ntNet）のAPIキーをサーバー側で隠す「中継役」が必要です。Cloudflare Pages Functions なら1日10万リクエストまで無料で利用できます。
- **Firebase Hosting + Cloud Functions だと有料プランが必要**: Firebase Hosting 自体は無料ですが、APIの中継に使う Cloud Functions は有料プラン（Blazeプラン＝使った分だけ課金される仕組み）への加入が必要です。

### 選ばなかった理由

- **Firebase Hosting**: ホスティング自体は無料ですが、Cloud Functions（サーバー側の処理）を使うには従量課金プランが必要です。APIキーを保護するための中継機能に費用がかかってしまいます。
- **Vercel**: 優秀なサービスですが、商用利用の場合は有料プランが推奨されます。
- **Netlify**: Cloudflare Pages と同等の機能がありますが、帯域幅に制限（月100GB）があります。

## 8. アプリの形態: PWA vs ネイティブアプリ（App Store） vs Webアプリ

アプリをどのような形で提供するかの選択です。

| 比較項目 | **PWA** | ネイティブアプリ（App Store） | 通常のWebアプリ |
|---------|---------|--------------------------|---------------|
| 費用 | 無料 | 年間約15,000円（Apple Developer登録） | 無料 |
| インストール方法 | ホーム画面に追加 | App Store からダウンロード | なし（URLアクセス） |
| 審査 | 不要 | Apple の審査が必要（数日〜数週間） | 不要 |
| アップデート | 即時反映 | 審査が必要 | 即時反映 |
| 全画面表示 | 可能 | 可能 | 不可（ブラウザのバーが表示） |
| ホーム画面のアイコン | 可能 | 可能 | 不可 |
| データの永続性 | 高い（ホーム画面追加時） | とても高い | 低い（7日で消える可能性） |
| カメラアクセス | 可能 | 可能 | 可能 |
| 開発の難易度 | ウェブ技術のみ | iOS専用言語（Swift）が必要 | ウェブ技術のみ |
| 子ども向けの適性 | 高い | とても高い | やや低い |

### 選んだ理由: PWA

- **費用ゼロで公開可能**: App Store に公開するには年間約15,000円の Apple Developer Program への登録が必要ですが、PWA なら無料です。
- **審査不要で即座に公開**: App Store のアプリは Apple の審査を通過する必要がありますが、PWA は開発者が好きなタイミングで公開できます。バグ修正も即座に反映できます。
- **5人の家族で使うなら十分**: App Store への公開は多くの人に使ってもらうためのものです。家族だけで使うアプリなら、URLを共有するだけでアクセスできる PWA が最適です。
- **ウェブ技術だけで開発可能**: ネイティブアプリは iOS 専用の言語（Swift）を学ぶ必要がありますが、PWA は React（ウェブ技術）だけで作れます。

### 選ばなかった理由

- **ネイティブアプリ**: 年間費用と審査の手間が、5人規模のアプリには見合いません。
- **通常のWebアプリ**: iOS Safari では7日間アクセスがないとデータが消える可能性があり、花の写真が失われるリスクがあります。PWA（ホーム画面に追加）ならこの問題を回避できます。

## 9. PDF生成: jsPDF vs html2canvas vs サーバーサイド生成

花の記録をPDFのアルバムにする技術の選択です。

| 比較項目 | **jsPDF** | html2canvas + jsPDF | サーバーサイド生成 |
|---------|-----------|--------------------|--------------------|
| 費用 | 無料 | 無料 | サーバー費用が必要 |
| 処理場所 | スマホの中 | スマホの中 | サーバー（インターネット上） |
| PDF の品質 | 良い（テキストがきれい） | 画像として変換（テキストが粗い） | とても良い |
| 日本語対応 | フォント追加が必要 | 問題なし | 問題なし |
| 開発の簡単さ | 普通 | 簡単 | 複雑（サーバーが必要） |
| 写真の取り扱い | 直接埋め込み可能 | HTML→画像→PDF の変換 | サーバーに写真を送る必要あり |
| 子ども向けの適性 | 高い | 高い | 高い |

### 選んだ理由: jsPDF

- **スマホの中だけで完結**: サーバーを使わずにPDFを生成できるため、追加のサーバー費用がかかりません。
- **写真を直接PDFに埋め込める**: IndexedDB（スマホ内）に保存された写真を直接PDFに埋め込めるため、サーバーに写真を送る必要がありません（プライバシー保護）。
- **テキストの品質が高い**: 花の名前や日付が「文字データ」としてPDFに埋め込まれるため、拡大しても鮮明です。

### 選ばなかった理由

- **html2canvas + jsPDF**: 画面のスクリーンショットをPDFにする方式です。手軽ですが、テキストが画像として保存されるためPDFの品質が劣ります。
- **サーバーサイド生成**: 高品質なPDFが作れますが、サーバーの費用がかかります。また、写真をサーバーに送信する必要があり、プライバシーの観点で問題があります。

## まとめ: この組み合わせが最適だった理由

Flower Shot の技術選定は、以下の3つの軸で判断しました。

### 軸1: コスト（すべて無料であること）

| 技術 | 費用 |
|------|------|
| React + TypeScript + Vite | 無料 |
| Tailwind CSS | 無料 |
| Firebase Auth + Firestore | 無料（5人規模なら） |
| Pl@ntNet API | 無料（500回/日まで） |
| Leaflet + OpenStreetMap | 完全無料 |
| Cloudflare Pages + Functions | 無料（帯域無制限） |
| jsPDF | 無料 |
| PWA | 無料 |
| **合計** | **月額0円** |

すべての技術を無料の範囲で利用できるため、家族で気軽に使い続けられます。

### 軸2: 子ども向けの適性

- **React + Tailwind CSS**: 大きなボタン、カラフルな配色、ひらがな中心のUIを自由にデザインできます。
- **Pl@ntNet API**: 花に特化したAIなので、子どもが花を撮影する体験にぴったりです。
- **PWA**: ホーム画面のアイコンをタップするだけで起動でき、子どもでも簡単に使えます。
- **Leaflet の地図**: シンプルな地図で、花を見つけた場所を直感的に振り返れます。
- **jsPDF のアルバム**: 印刷して実物のアルバムとして残せるため、デジタルに親しみのない子どもにも成果が伝わります。

### 軸3: 開発・運用の簡単さ

- **Firebase（Auth + Firestore）の一体運用**: ログインとデータ保存が同じサービス内で完結し、セキュリティ設定も簡単です。
- **Cloudflare Pages の自動デプロイ**: プログラムを更新するだけで自動的にアプリが更新されます。
- **サーバー管理不要**: すべてクラウドサービスに任せているため、サーバーの管理や監視が不要です。
- **画像のローカル保存（IndexedDB）**: 写真をクラウドに送らないことで、ストレージの管理やバックアップの仕組みが不要になりました。

### 最終結論

Flower Shot の技術スタックは、**「無料」「子ども向け」「簡単に運用できる」** という3つの条件をすべて満たす、最適な組み合わせです。

特に、**Firebase（認証 + データベース）と Cloudflare Pages（公開 + APIプロキシ）の組み合わせ** が今回の決め手でした。Firebase だけではAPIプロキシに費用がかかり、Cloudflare だけでは認証とデータベースを別途用意する必要があります。両者を組み合わせることで、それぞれの無料枠を最大限に活用し、費用ゼロでの運用を実現しています。

